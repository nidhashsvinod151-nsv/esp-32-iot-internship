# CoAP Overview — CON, NON, and OBSERVE

## What is CoAP
CoAP (Constrained Application Protocol) is a lightweight, REST-like protocol
that runs over UDP instead of TCP. It uses familiar verbs (GET, POST, PUT,
DELETE) but with tiny binary headers (~4 bytes) instead of HTTP's verbose
text headers, and no connection handshake — making it ideal for battery-
powered, constrained devices that wake briefly, send data, and sleep again.

## CON (Confirmable)
A Confirmable message requires the receiver to send back an acknowledgment
(ACK). This is CoAP's lightweight equivalent of reliable delivery — used
when the sender genuinely needs to know the message arrived, such as a
critical command or alert. If no ACK is received, the sender retransmits
with a backoff, similar in spirit to TCP retransmission but far lighter.

## NON (Non-confirmable)
A Non-confirmable message is fire-and-forget: no acknowledgment is expected
or sent back. This is the most power-efficient mode and is well suited to
routine periodic sensor readings where losing an occasional packet is
acceptable, since another reading will follow shortly after.

## OBSERVE
The OBSERVE option lets a client subscribe to a resource once and receive
automatic push updates whenever the value changes, without needing a
persistent broker connection like MQTT requires. The server simply sends
a new notification each time the observed value updates. This gives CoAP
an MQTT-like publish/subscribe pattern, but without the overhead of
maintaining an always-open broker session.

## Why This Matters for Zelbytes
The current DHT22 node is mains-powered, so MQTT's persistent session cost
is not a concern. CoAP becomes relevant for future battery-powered sensors
(see Day 14, soil moisture on CR2032) that cannot afford to keep a radio
connection open continuously and must wake, transmit, and sleep to survive
months on a coin cell.
